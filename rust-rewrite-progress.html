Update — March 15, 2026 (Session 2)

8 fixes applied, 2 audit cycles completed, performance re-measured at steady state. Ad merging, heuristic refinement, and oneshot refinement all implemented.

Bugs Fixed This Session

⚠ SQLite session store — replaced in-memory MemoryStore with SQLite-backed store. Sessions now persist across container restarts.
⚠ NULLS LAST (5 queries) — replaced PostgreSQL-only NULLS LAST with SQLite-compatible ORDER BY release_date IS NULL, release_date DESC in all post-ordering queries.
✓ base_url HTTPS support — added BASE_URL / PODLY_BASE_URL env var support so RSS feed URLs use the correct scheme behind a reverse proxy (was hardcoded http://).
✓ Ad segment merging — added proximity-based merging using min_ad_segment_separation_seconds from DB settings. Segments within the gap threshold are merged into groups before audio cutting.
✓ Heuristic refinement fallback — when LLM boundary refinement fails, falls back to pattern-based refinement using intro/outro keywords ("brought to you", "sponsor", ".com", "use code", etc.).
✓ Oneshot refinement — boundary refinement now applies to all ad detection strategies including oneshot (was previously skipped).

Assessment: Is Performance as Expected?

Memory (19x less): Expected for Rust vs Python. Python runs 2 separate processes (Flask app + background writer), each loading the full Python runtime, SQLAlchemy ORM, and dependency tree. Rust runs one async binary with zero-copy deserialization. The 82 MiB steady-state includes SQLite page cache, connection pools, and session data — reasonable for a 311 MB database. Python's 1.4 GiB is typical for a Flask+Gunicorn+SQLAlchemy stack with multiple workers.

Image size (14x smaller): Expected. Python image includes the full Python runtime, pip packages, whisper models, ffmpeg, and ML dependencies. Rust produces a single statically-linked binary (~21 MB) in a minimal container.

CPU (both ~0%): Expected at idle. Processing throughput is dominated by external API calls (Whisper, LLM) and ffmpeg — backend language has minimal impact on total job time.

I/O (49x less): Python's 3.1 GB block I/O write over 6 days reflects ORM journaling overhead, logging, and the separate writer process. Rust's 63 MB in ~1 hour suggests minimal write amplification from direct SQL queries.

Notable Fixes

Security

⚠ Aggregate feed endpoint had no auth check — any user could access another user's RSS feed
⚠ SQL table name mismatch (user_feed vs feed_supporter) would have caused runtime crashes
⚠ RSS feed served unprocessed posts (missing processed_audio_path IS NOT NULL filter)

Processing Pipeline

✓ Streaming download — prevents OOM on large podcast episodes (was loading entire file into memory)
✓ User-Agent + Referer headers — fixes downloads from CDNs that reject bare requests
✓ Post-level metadata in classifiers — was using feed title/description instead of episode title/description
✓ fade_ms from config — was hardcoded 50ms, DB default is 3000ms (60x difference!)
✓ Whisper model call preservation — reprocessing now reuses existing transcripts correctly

API Compatibility

✓ Discord OAuth consent prompt retry logic with interaction_required handling
✓ Stripe checkout URL field renamed from "url" to "checkout_url"
✓ Job priority computed dynamically (running=2, pending=1) instead of hardcoded 0
✓ bcrypt fallback with transparent Argon2 re-hash on login (Python password compatibility)
✓ 90+ API response shape, status code, and error message fixes across all endpoints
✓ Config PUT now handles all 16 LLM/processing fields (was missing 6)

Known Remaining Differences

These are documented, accepted differences — either architectural choices or deferred low-impact items.

Architectural (By Design)

  • post_debug returns JSON (Python returns HTML) — frontend prefers JSON
  • add_feed returns JSON (Python returns 302 redirect) — frontend sends AJAX
  • search_feeds uses PodcastIndex API (Python uses iTunes) — same result format

Pipeline (Remaining)

  • Content-aware ad merging (keyword/sponsor detection) — Rust uses proximity-only merging
  • No token rate limiting for LLM API calls
  • No max_completion_tokens for newer OpenAI models
  • No pre-reprocess snapshot creation
  • No input token count validation/trimming for oversized prompts

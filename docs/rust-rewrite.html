Podly Rust Migration

Engineering assessment of a complete Python/Flask to Rust/Axum backend rewrite

March 2026 · 11,884 lines of Rust · 36 documented decisions · 8 audit cycles

Should This Migration Happen?

The Python backend runs two processes consuming 1.4 GiB of RAM to serve what is essentially a CRUD API with background job processing. The Rust version does the same work in 82 MiB. For a self-hosted application running on resource-constrained hardware alongside other services, this is a meaningful difference.

The more significant benefit is operational: a single statically-linked binary with no Python runtime, no pip dependency tree, no Gunicorn worker management. Deployment is "copy one file." The Docker image drops from 8.66 GB to 603 MB.

The cost is maintainability. The Python codebase is more approachable to contributors. Rust's learning curve is real. The rewrite was done by AI in three days; a human maintaining it needs to be comfortable with async Rust, raw SQL, and the Axum middleware stack.

Recommendation: Proceed, with investment in testing. The performance gains are real and the code is structurally sound. The remaining work is validation, not feature development.

Why Not Use the Stripe SDK?

A community crate (stripe-rs) exists. The rewrite uses raw HTTP instead: 6 helper functions calling Stripe's REST API with Basic Auth, plus 35 lines of webhook HMAC verification.

This loses type-safe request/response structs, automatic retry logic, and forward compatibility with Stripe API changes. It gains a smaller dependency surface for the 6 endpoints actually used. Acceptable for the current simple billing model; would become a liability if billing complexity grows.

Why sqlx Instead of an ORM?

88 query functions, 911 lines of hand-written SQL. No compile-time schema validation. If a column is renamed, it's a runtime error.

The benefit is explicit control: every query is visible, there are no hidden N+1 problems, and the Python codebase already used explicit SQL through SQLAlchemy. The sqlx::query!() compile-time macro could be adopted incrementally without rewriting queries — same SQL strings, just checked at build time.

Why PodcastIndex Instead of iTunes?

PodcastIndex is an open, community-maintained podcast index with authenticated API access. iTunes is unauthenticated but technically undocumented and has been unreliable. The response format is normalized to the same shape, so the frontend sees no difference. Niche or region-specific podcasts may have different availability.

LLM Integration: genai vs litellm

Python's litellm supports 100+ LLM providers behind a single call. Rust's genai crate natively supports OpenAI, Anthropic, Gemini, and Groq. If the user configures a provider genai doesn't support, classification will fail. This covers >95% of practical use cases but is a regression in provider breadth.

Ad Segment Merging: Simplified

Python's AdMerger does three passes: proximity grouping, content-aware keyword extraction (URLs, promo codes, brand names), and weak-group filtering. Rust does proximity grouping + short-segment filtering only. The content-aware pass was deferred because it requires transcript segment text and ORM relationships not easily available at the merge step. In multi-sponsor episodes, this may produce slightly different (but valid) audio cuts.

Before Merging

Must-Have

  • End-to-end processing test. Process at least one real podcast episode and verify the output audio.
  • Integration tests for the critical path. Login, add feed, process episode, serve RSS.
  • Docker with local whisper (if local transcription is a requirement for any users).

Should-Have

  • Compile-time query checking. Adopt sqlx::query!() for critical queries to catch schema mismatches at build time.
  • ARM64 Docker build (if ARM deployment is needed).
  • Stripe SDK evaluation. Assess whether stripe-rs is mature enough to replace raw HTTP.
  • LLM provider documentation. Clearly list supported providers and configuration patterns.

Nice-to-Have

  • Content-aware ad merging (keyword/sponsor detection)
  • Token rate limiting for LLM API calls
  • max_completion_tokens support for newer OpenAI models
  • Input token validation to prevent oversized prompts

Data Migration

A standalone binary (migrate_legacy) copies all 12 tables from a Python-era SQLite database to the Rust schema. It preserves all IDs (maintaining foreign key integrity), handles minor schema differences between the two backends, and supports --dry-run mode.

What migrates: feeds, posts, users, transcript segments, model calls, identifications, feed memberships, access tokens, job history, and all settings tables.

What doesn't migrate: user sessions (different session store implementation), temporary audio files (only paths stored).

Risk: No transaction wrapping. If migration fails partway (disk full, schema mismatch), the destination database is inconsistent with no rollback. Always back up the destination database first.

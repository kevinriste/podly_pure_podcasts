Notable Fixes

Security

  • Aggregate feed endpoint had no auth check — any user could access another user's RSS feed
  • SQL table name mismatch (user_feed vs feed_supporter) would cause runtime crashes
  • RSS feed served unprocessed posts (missing processed_audio_path IS NOT NULL filter)
  • password_hash excluded from JSON serialization via #[serde(skip_serializing)]

Processing Pipeline

  • Streaming download prevents OOM on large episodes (was loading entire file into memory)
  • User-Agent + Referer headers for CDN compatibility (acast, etc.)
  • Post-level metadata in classifiers (was using feed title instead of episode title)
  • fade_ms from DB config (was hardcoded 50ms, default is 3000ms)
  • Ad segment merging with configurable gap threshold from DB
  • Short-segment filtering with configurable min length from DB
  • Last-segment extension to audio end when within gap threshold
  • Heuristic refinement fallback when LLM parsing fails
  • ffmpeg simple fallback when complex filter graph fails
  • Boundary refinement applied to all strategies including oneshot

Infrastructure

  • SQLite-backed session store (sessions persist across restarts)
  • 5 queries fixed: PostgreSQL-only NULLS LAST replaced with SQLite-compatible ordering
  • BASE_URL env var for RSS URLs behind HTTPS reverse proxy
  • bcrypt fallback with transparent Argon2 re-hash on login
  • 90+ API response shape, status code, and error message fixes
  • Config PUT handles all 16 LLM/processing fields

Accepted Differences & Remaining Gaps

These are documented, reasoned choices — either architectural improvements or deferred low-impact items. See DECISIONS.md for rationale.

Architectural (By Design)

  • POST /feed returns 201 JSON (Python returns 302 redirect) — frontend sends AJAX
  • search_feeds uses PodcastIndex API (Python uses iTunes) — same result format
  • post_debug returns JSON (Python returns HTML) — frontend prefers JSON
  • Ad merging is proximity-only; Python adds keyword/sponsor content analysis (DECISION-034)
  • LLM calls use genai crate; Python uses litellm (DECISION-020)

Pipeline (Deferred)

  • No proactive token rate limiting (reactive 429 backoff only)
  • max_tokens used instead of max_completion_tokens for newer OpenAI models (DECISION-036)
  • No pre-reprocess snapshot creation
  • No input token count validation/trimming for oversized prompts

Infrastructure

  • JobsManagerRun lifecycle not created by Rust (reads existing Python-era runs)

Not Yet Tested End-to-End

  • Full process/reprocess lifecycle (download, transcribe, classify, refine, merge, cut)
  • Scheduler/background job execution
  • Billing/Stripe webhook handling
  • Discord OAuth flow (config endpoint tested only)

Performance Assessment

Memory (19x less): Expected. Python runs 2 processes (Flask app + background writer), each loading the full Python runtime, SQLAlchemy ORM, and dependency tree. Rust runs one async binary with zero-copy deserialization. 82 MiB steady-state includes SQLite page cache, connection pools, and session data — reasonable for a 311 MB database.

Image (14x smaller): Expected. Python image includes Python runtime, pip packages, whisper models, ffmpeg, and ML dependencies. Rust produces a single statically-linked binary (~21 MB) in a minimal container.

CPU (both ~0% idle): Expected. Processing throughput is dominated by Whisper API, LLM API, and ffmpeg — backend language has minimal impact on total job time.

I/O (49x less): Python's 3.1 GB over 6 days reflects ORM journaling, logging, and separate writer process. Rust's 63 MB in ~1 hour suggests minimal write amplification from direct SQL.
